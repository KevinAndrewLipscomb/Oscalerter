000-README


Abstract

This document describes additional steps that you must perform, after checking out the OscalertConsoleApp project from its repository, to achieve a clean compilation or to execute the application. This document does not cover how to meet other prerequisites, such as setting creating and populating the database schema.

1 Recommended methods

The use of NTFS reparse points is recommended in this document because it allows versioning operations to take place without undesirable impact to project environments, and without violating versioning rules. If you simply exported (or downloaded as a ZIP) OscalertConsoleApp from its origin repository and you do not intend to perform subsequent updates or any other versioning operations on it (not recommended), you can simply make copies instead of NTFS reparse points.

2 Establish hooks to adjunct projects

2.1 Hook up the KiDotnetFoundation project (only if compiling)

OscalertConsoleApp uses object code from the KiDotnetFoundation adjunct project. The KiDotnetFoundation is used by multiple production-eligible projects and its source code is version controlled independently from those production-eligible projects. The object code is already provided as OscalertConsoleApp/bin/ki.dll to enable execution of OscalertConsoleApp in a production environment, but to hook up the KiDotnetFoundation project for the sake of a clean compile of OscalertConsoleApp, do the following:

  1. Check out the https://kveo-it-project.svn.sourceforge.net/svnroot/kveo-it-project/trunk/foundation/dotnet/ki project. Put it in a KiDotnetFoundation folder. Make sure the KiDotnetFoundation folder is not inside the OscalertConsoleApp folder tree.

  2. Create a junction reparse point named ki inside the OscalertConsoleApp/lib folder that points to the KiDotnetFoundation folder. You should end up with a folder inside the OscalertConsoleApp/lib folder called ki. You can make junction points with hermann schinagl’s link shell extension tool or, on Vista-based systems (including Windows 7, Windows Server 2008, and later systems), the prototype commands would be:

    cd OscalertConsoleApp/lib
    mklink /j ki KiDotnetFoundation

3 Put appropriate config files in the appropriate application folders

OscalertConsoleApp will not run without OscalertConsoleApp.exe.config and unpublished.config files in its bin folder. Because the contents of those files are environment-specific, and we wish to keep the rest of the OscalertConsoleApp folder tree environment-neutral, those config files are kept in the OscalertConsoleApp/config subfolder tree. To put config files that are appropriate for your environment into the OscalertConsoleApp/bin folder, do the following:

  1. Copy the OscalertConsoleApp/config/template-unpublished.config-template file to an appropriate subfolder of OscalertConsoleApp/config, rename it to unpublished.config, and revise its contents appropriately. Do not add any unpublished.config file to any public repository.

  2. Copy the appropriate unpublished.config file from within the OscalertConsoleApp/config folder tree to the OscalertConsoleApp/bin folder.

  3. Copy the appropriate App.config file from within the OscalertConsoleApp/config folder tree to the OscalertConsoleApp/bin folder and rename it to OscalertConsoleApp.exe.config.

From the Windows command line, the prototype commands would be:

cd OscalertConsoleApp
mklink bin/unpublished.config config/{appropriate-path}/unpublished.config
copy config/{appropriate-path}/App.config bin/OscalertConsoleApp.exe.config
